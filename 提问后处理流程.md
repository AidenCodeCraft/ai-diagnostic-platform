# AI Diagnostic Platform 整体诊断流程

## 一、用户侧操作流程

用户（开发、管理员、用户）：

- **输入**：例如 "设备启动后偶发黑屏，重启恢复，最近出现 3 次。"
- **上传**：
  ```
  device_log_20260717.txt
  kernel.log
  dmesg.log
  serial.log
  crash_dump.log
  ```
- **点击**：
  ```
  发送
  ```

### 整体流程

```
用户
  |
  | 1. 提交问题描述
  | 2. 上传日志文件
  ↓

Web/Desktop/Mobile 客户端
  ↓
API Gateway
  ↓
诊断任务系统
  ↓
日志处理流水线
  ↓
AI分析系统
  ↓
诊断报告
  ↓
用户查看结果
```

---

# 二、后台详细处理流程

---

## Step 1：接收诊断请求

**对应模块**：`backend/api/diagnostic-service`

**接口**：例如 `POST /api/v1/diagnosis/create`

**请求**：

```json
{
    "description": "设备启动后黑屏，重启恢复",
    "project_id": 10001
}
```

**同时上传**：multipart/form-data -- log_file: xxx.log

后台创建任务，数据库 `diagnosis_tasks`：

| 字段 | 值 |
| --- | --- |
| id | 10086 |
| status | uploaded |
| description | 启动黑屏 |
| user_id | 001 |
| project_id | device-A |

状态：`CREATED`

---

## Step 2：日志文件保存

**模块**：Log Management Service

**流程**：

```
上传文件
↓
文件校验
↓
计算Hash
↓
保存
↓
生成Log记录
```

**目录**：例如 `data/raw/device-A/2026-07-17/boot.log`

**数据库 logs 表**：id, filename, path, size, hash, project_id, status

状态：`UPLOADED`

---

## Step 3：日志预处理

进入：Log Parser Engine（第 005 阶段）

作用：把文本日志转换为结构化事件。

**原始**：

```
[ 123.455 ] mmc0: timeout waiting for interrupt
```

**转换**：

```json
{
    "time": "123.455",
    "module": "storage",
    "level": "ERROR",
    "message": "timeout waiting for interrupt"
}
```

### 3.1 日志清洗

去除：无意义打印、重复信息、时间噪声

### 3.2 日志分类

识别：kernel, driver, application, network, hardware

### 3.3 错误提取

发现：kernel panic, segmentation fault, usb disconnect, wifi timeout, mmc error

生成：

```json
[
    { "type": "kernel_panic", "severity": "critical" },
    { "type": "storage_timeout", "severity": "high" }
]
```

---

## Step 4：错误规则分析（Rule Engine）

这是非常重要的一层。不要一开始全部依赖大模型。

**架构**：

```
日志
↓
规则库
↓
确定性分析
↓
AI分析
```

**规则示例**：

```yaml
name: MMC_TIMEOUT
pattern: "mmc.*timeout"
solution: 检查EMMC供电和时钟
severity: HIGH
```

**匹配**：日志 `mmc timeout`

**输出**：

```json
{
    "issue": "EMMC通信异常",
    "confidence": 0.9
}
```

---

## Step 5：知识库检索（RAG）

如果规则不能确定，进入 RAG 系统。

**流程**：

```
错误信息
↓
Embedding模型
↓
向量数据库
↓
搜索相似案例
```

**知识库来源**：芯片 Datasheet, 驱动文档, Git Issue, 历史 Bug, 测试报告, 维修记录

**搜索**：`BMA421 init failed`

**找到**：历史问题：I2C pull-up 不足导致初始化失败。解决：增加上拉电阻

**返回**：相关经验 TOP 5

---

## Step 6：Agent 智能分析

进入核心：Diagnostic Agent

Agent 收到三个输入：

### 输入 1：用户描述

```
启动黑屏
```

### 输入 2：日志分析结果

```json
{
    "error": "gpu init failed",
    "time": "boot stage"
}
```

### 输入 3：RAG 知识

```
历史GPU初始化失败案例
```

**Agent 推理**：

```
用户说：启动黑屏
日志：gpu driver init timeout
知识：该芯片历史存在DDR初始化异常

推理：
可能原因：
1. DDR初始化失败
2. GPU驱动加载失败

建议：
检查DDR timing、kernel config
```
